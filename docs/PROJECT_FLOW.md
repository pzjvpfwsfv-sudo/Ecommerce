# PROJECT_FLOW

## 第 0 章目标

第 0 章不追求把所有中间件一次性跑起来，而是先回答三个问题：

1. 这个项目到底要解决什么问题
2. 最小可运行版本应该先保留哪些组件
3. 后面每一章为什么要加新组件，而不是为了“看起来高级”

## 项目要解决的问题

我们要模拟一个电商平台，把用户行为、订单、支付等数据变成可以实时查看、可以做经营分析、可以被 AI 助手问数的数据平台。

真正想训练出来的能力有三类：

- 实时链路能力：数据怎么采集、传输、计算、落库
- 数据服务能力：指标怎么暴露给看板和接口
- 工程化能力：怎么压测、调优、定位问题、沉淀证据

## 最小主链路解释

### 1. 数据生成器

职责：

- 模拟用户浏览、点击、加购、下单、支付等行为
- 让后续 Kafka/Flink/Doris 有真实输入

为什么不是直接手写几条测试数据：

- 面试里说不出吞吐、乱序、重复、异常数据
- 没法支撑后面的压测和调优故事

### 2. Kafka

职责：

- 暂存和转发实时事件流
- 让生成器和 Flink 解耦

为什么不直接写数据库：

- 上游流量一大就容易耦合和阻塞
- 实时计算更适合从消息队列消费事件流

### 3. Flink

职责：

- 负责清洗、去重、窗口聚合、状态管理
- 计算 PV、UV、GMV、转化率、Top 商品等实时指标

为什么它是中枢：

- 后续 watermark、checkpoint、state、反压、并行度都在这里体现技术含量

### 4. Doris

职责：

- 给看板提供低延迟分析查询

为什么不是让看板直接查 Flink 或原始日志：

- 看板要的是稳定、快、面向分析的结果集

### 5. FastAPI

职责：

- 把 Doris 或后续聚合结果封装成前端接口
- 后面还能挂 AI 指标助手

### 6. 看板

职责：

- 展示核心实时指标
- 给项目演示、README 截图和简历提供可见成果

## 第 0 章产出

第 0 章结束时，我们应该具备这些东西：

- 一个干净的项目目录
- 一份清楚的主链路说明
- 一份知道先后顺序的章节路线
- 对每个组件“为什么存在”的理解

## 当前目录职责

### `docs/`

- 放项目说明、数据契约、调优记录、压测结果、面试材料

### `infra/`

- 放 `docker-compose.yml`、环境变量模板和中间件配置

### `generators/`

- 放行为日志和业务数据模拟器

### `jobs/`

- 放 Flink SQL、Flink 作业代码和任务配置

### `services/`

- 放 FastAPI、后续 AI 服务和数据接口层

### `dashboards/`

- 放前端看板

### `scripts/`

- 放初始化、启动、自检脚本

## 下一章会做什么

第 1 章我们会进入真正的工程初始化，先把下面这些东西补起来：

- Git 仓库初始化
- 基础目录和占位文件
- `infra/` 下的 Compose 骨架
- 后续开发约定

## 面试话术

如果面试官问“为什么你的项目不是一上来就 Iceberg、CDC、Trino、AI 全上”，你可以这样回答：

> 我先按最小主链路做 MVP，把数据从生成、传输、计算、查询到展示先跑通。这样每一层都有验证标准，后面再逐步加入湖仓、CDC、即席查询和 AI 问数，能够清楚说明每个组件解决的实际问题，而不是为了堆技术栈。
