# 第 1 章分阶段 Compose 基础设施设计

## 1. 设计目标

第 1 章的目标不是把整个平台一次性拉起，而是先把基础设施骨架搭稳，具体包括：

- 建立统一的 `docker-compose.yml` 作为容器编排入口
- 建立统一的环境变量模板，避免配置散落在多个文件里
- 先为 Kafka 和 API 服务边界预留清晰位置
- 为 Flink、Doris、MinIO 以及后续的 KRaft 演进保留扩展空间

## 2. 方案选择

本章采用的是 `分阶段 Compose` 路线，而不是一次性启动全部组件。

### 当前阶段启用

- ZooKeeper
- Kafka
- API 占位服务

### 后续阶段逐步接入

- Flink
- Doris
- MinIO / Iceberg
- Trino
- AI 指标分析助手

## 3. 为什么这样设计

如果第一章就把所有中间件一起拉起，很容易在环境、机器资源和配置问题上卡住，影响节奏。
分阶段 Compose 的好处是：

- 更适合当前“边学边做”的推进方式
- 更接近真实工程中的渐进式建设思路
- 更方便后续拆解成面试时能讲清楚的阶段成果

## 4. 预留的架构演进

Kafka 当前先采用 ZooKeeper 模式，目的是降低首次搭建复杂度，尽快把消息入口搭起来。

后续会明确安排一次 `ZooKeeper -> KRaft` 升级，把它做成一段可以用于面试的“架构演进”故事，重点包括：

- 为什么项目前期先选 ZooKeeper
- 为什么后续迁移到 KRaft
- 迁移时改了哪些配置
- 如何验证迁移前后服务行为一致

## 5. 本章边界

### 本章要完成的内容

- Compose 骨架
- 环境变量模板
- 按组件拆分的配置目录
- README 中的阶段说明

### 本章暂时不做的内容

- Flink 实时作业
- Doris 查询层
- MinIO / Iceberg 湖仓层
- 数据生成器
- 真正的 FastAPI 业务逻辑

## 6. 本章预期结果

第 1 章结束后，仓库应该从“只有项目目录”升级为“具备明确基础设施边界和扩展路线的容器化数据平台骨架”，并且能够为第 2 章的 `数据生成器 -> Kafka` 链路开发做好准备。

## 7. 后续演进回写

第 1 章当时预留的 `ZooKeeper -> KRaft` 路线，已经在第 7 章正式落地。

现在回看，第 1 章先用 ZooKeeper 起步仍然是合理的，因为它帮助项目更快跑通了最早期的消息入口和 Flink 主链路；但后续随着 Doris、Iceberg、Hive Metastore、Trino 一步步接入，Kafka 的控制面问题也越来越不能只靠脚本层兜底。

最终落地结果是：

- ZooKeeper 已从 Compose 中移除
- Kafka 已切换到 KRaft
- 采用 `controller + broker` 双角色拓扑
- 对外入口仍保持 `localhost:9092`
- 容器内入口仍保持 `kafka:29092`

这说明第 1 章不只是搭了一个“临时骨架”，而是真的为后续架构演进留出了正确接口和叙事空间。
