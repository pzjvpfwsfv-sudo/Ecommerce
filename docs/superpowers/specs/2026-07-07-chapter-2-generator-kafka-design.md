# 第 2 章数据生成器接入 Kafka 设计

## 1. 设计目标

第 2 章的目标是把“可持续产生业务事件的数据源”补上，让平台第一次具备真实的实时数据入口，具体包括：

- 设计一套最小可用的电商用户行为事件结构
- 在本地持续生成模拟行为数据
- 把事件稳定写入 Kafka 的 `user_behavior_events` topic
- 为第 3 章 Flink 实时计算提供稳定输入

## 2. 方案选择

本章采用的是 `Python 数据生成器 + Kafka Producer` 路线，而不是一开始就引入更复杂的数据采集组件。

### 当前阶段生成的事件类型

- `view`
- `click`
- `cart`

### 当前阶段事件字段

- `event_id`
- `user_id`
- `product_id`
- `event_type`
- `event_time`
- `channel`
- `device_type`
- `page_id`

## 3. 为什么这样设计

这一章的关键不是把事件模型一次性做全，而是先保证“有连续、稳定、可观察的数据流”。

先用 Python 做生成器有几个明显好处：

- 开发和调试成本低，适合快速验证 Kafka 链路
- 事件结构清晰，便于后续扩展订单、支付、退款等主题
- 更适合当前项目的分章推进方式，先把主链路跑通，再逐步复杂化

相比一开始就接入 CDC、埋点 SDK 或多 topic 方案，这种设计更轻，也更容易沉淀成一段能讲清楚的项目故事。

## 4. 运行方式设计

本章的运行入口保持尽量简单：

- 通过 `generators/run_generator.py` 持续生成并发送事件
- 通过环境变量控制 Kafka 地址、topic 和发送间隔
- 每次发送后在控制台打印一条简洁日志，方便观察链路状态

当前默认配置：

- Kafka 地址：`localhost:9092`
- Topic：`user_behavior_events`
- 发送间隔：`1` 秒

## 5. 本章边界

### 本章要完成的内容

- 随机用户行为事件生成器
- Kafka Producer 封装
- 单次发送的最小测试覆盖
- 可直接运行的本地入口脚本

### 本章暂时不做的内容

- 订单、支付、库存等更多业务事件
- 多 topic 路由
- 死信队列、重试队列和复杂容错
- 数据质量校验平台
- 最终项目版的统一数据回放体系

## 6. 与后续章节的衔接

第 2 章的数据结构会故意保持“足够用但不过度设计”。

这样做的原因是：

- 第 3 章只需要先验证 Kafka -> Flink SQL 的主链路
- 第 4 章开始接 Doris 和 API 时，指标侧会进一步反推我们需要补哪些事件
- 后续如果切到 DataStream API，再补复杂事件逻辑会更自然

换句话说，本章更像是在搭“实时事件输入底座”，而不是提前完成整套生产级埋点体系。

## 7. 预期结果

第 2 章结束后，仓库应该从“只有 Kafka 基础设施”升级为“能够持续向 Kafka 注入电商行为事件”的状态。

只要 Kafka 启动正常，我们就应该能够看到：

- 生成器持续打印发送日志
- Kafka topic 中不断累积新的事件消息
- 第 3 章可以直接消费这些消息做实时计算
